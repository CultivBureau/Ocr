# BureauOCR — Frontend Specification

**Product:** BureauOCR (PDF/Word → structured travel-document editing and PDF output)  
**Repository:** `Pdf-Converter` (Next.js App Router)  
**Companion backend spec:** `Pdf-Converter-backend/.spec.md`  
**Document type:** Living specification — update this file whenever extraction contracts, routes, or UX around OCR change.

**Last updated:** 2026-04-12

---

## 1. Purpose of this document

This file is the single source of truth for **how the frontend integrates with document upload, OCR/structuring, and downstream editing**. It describes the **current** behavior and the **planned** backend enhancement (Analyzer → Router → Modules → GPT refinement) so UI and types can evolve in lockstep with the API.

---

## 2. High-level product flow (frontend)

1. User authenticates (JWT in cookies; `Authorization: Bearer` on API calls).
2. User uploads a document (**PDF**, **DOCX**, or legacy **DOC**) via `POST /upload/`.
3. Frontend requests structured extraction via `POST /extract/structured` (blocking) or async `POST /extract/structured/start` + polling `GET /extract/jobs/{job_id}` (available in `PdfApi`; primary UX may still use blocking extract depending on page).
4. Backend returns **v2 separated structure** (`SeparatedStructure`): AI-generated sections/tables, empty user elements, layout order, `meta`, optional **component suggestions** (flights, hotels, transport).
5. User edits structure, approves suggestions, adds airplane/hotel/transport blocks, exports PDF via backend Playwright generation.

The frontend does **not** run OCR locally; it only orchestrates API calls and renders JSON.

## 4. Current implementation map

### 4.1 API surface (`PdfApi.ts`)

| Function | HTTP | Purpose |
|----------|------|---------|
| `uploadFile` / `uploadFileWithProgress` | `POST /upload/` | Multipart upload; returns `file_path` for extraction |
| `extractStructured` | `POST /extract/structured` | **Blocking** full pipeline; returns `SeparatedStructure` |
| `startExtraction` | `POST /extract/structured/start` | Async job start; returns `job_id` |
| `getExtractionJob` | `GET /extract/jobs/{job_id}` | Poll until `completed` / `failed` |
| `generatePDFWithPlaywright` | `POST /api/pdf/generate` | Server-side PDF from saved document |

Client-side limits (aligned with backend messaging): **20 MB** upload cap in `uploadFileWithProgress`; backend enforces page limits and plan quotas.

### 4.2 Core types (`ExtractTypes.ts`)

- **`Section` / `Table`:** Generated blocks with `id`, `title`, `content` / `columns`+`rows`, `order`, etc.
- **`UserElement`:** User-created `airplane` | `hotel` | `transport` with typed `data` payloads.
- **`SeparatedStructure` (v2):**
  - `generated: { sections, tables }`
  - `user: { elements: UserElement[] }`
  - `layout: string[]` — ordered IDs (`gen_sec_*`, `gen_tbl_*`, `user_*`)
  - `meta: Record<string, unknown>` — includes `structure_version`, counts, `extraction_method`, timestamps
  - `suggestions?: ComponentSuggestion[]` — optional post-extraction hints

IDs from the API are normalized server-side to **`gen_sec_*`** and **`gen_tbl_*`** prefixes.

### 4.3 State (`extractSlice.ts`)

Central reducer for upload/extract state: separated structure, legacy compatibility, maps for quick lookup, processing flags. New extraction features should extend this slice only when the **user-visible contract** changes.

### 4.4 Upload hook (`hooks/useUpload.ts`)

Default path: **`uploadFile` → `extractStructured(file_path)`** (blocking). Async job APIs exist for long-running or polling UX; wire them where product requires non-blocking flows and proxy timeouts are a concern.

### 4.5 Document editor surface (non-exhaustive)

- **Routing:** e.g. `(routes)/pdf/document/[id]/page.tsx` for persisted documents.
- **Rendering:** `StructureRenderer`, section/table templates, airplane/hotel/transport templates.
- **Suggestions UI:** previews for approved component suggestions; types in `ComponentSuggestion`.

## 5. Contract: what the frontend assumes today

1. **Success shape:** At least one of `generated.sections` or `generated.tables` is populated for a non-empty document (see `useUpload` validation).
2. **Layout:** `layout` lists IDs in display order; frontend reconciles with `generated` + `user.elements`.
3. **Meta:** `meta.structure_version === 2` for current pipeline; `meta.extraction_method` describes backend path (e.g. `google_document_ai_gpt` for PDF).
4. **Suggestions:** Best-effort; failures on the backend do not fail extraction.

## 6. Planned backend enhancement (frontend impact)

Backend will move from **“one big GPT structuring pass”** to:

**PDF → Document AI → Analyzer (cheap) → Router (logic) → Modules (core) → GPT (refinement only) → final JSON.**

### 6.1 Expected additions (to design against)

| Concern | Likely change |
|---------|----------------|
| **Routing module** | New `meta` fields, e.g. `extraction_module` or `bureau_module` with enumerated module id (see backend spec). |
| **Analyzer output** | Flags such as `hasFlights`, `hasTransfers`, `hasHotelsBlocks`, `hasDayParagraphs`, `hasPricingTable`, `hasStrictTableFormat` — may be exposed for debugging or UI badges (optional). |
| **GPT role** | Smaller, refinement-only payloads; frontend still receives the same **v2** envelope unless we version again (`structure_version: 3`). |
| **UI** | Optional: module-specific templates, warnings, or empty states when a module detects narrative vs table-heavy itineraries. |

### 6.2 Frontend tasks when backend ships

- Extend `SeparatedStructure.meta` typing (or narrow with Zod) for new keys.
- If `layout` or element shapes change per module, update `StructureRenderer` and migrations (`structureMigration.ts`).
- Update copy/timeouts if refinement reduces latency or changes suggestion timing.
- Keep **`PdfApi.extractStructured` response type** as the single contract; avoid duplicating backend enums in many files — prefer one exported type module.
